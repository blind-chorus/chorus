# lovable-export

> **Designing in Lovable?** Read [`PROMPTING.md`](PROMPTING.md) first — it has the copy-paste preamble and the intent → component map that keep Lovable on Chorus components even when the prompt is vague about visual style. [`AGENTS.md`](AGENTS.md) is the matching contract the model is expected to follow; the seed at [`src/routes/index.tsx`](src/routes/index.tsx) is the canonical composition example.

Mirror of the [chorus-lovable-template-v1](https://github.com/blind-dsai/chorus-lovable-template-v1) layout, generated from this repo so we can keep the chorus monorepo as the source of truth while still using Lovable's GitHub-synced editor.

## How it works

- Baseline files (configs, `src/router.tsx`, `src/routes/__root.tsx`, error handling, etc.) are kept **byte-equivalent** to the Lovable template so the editor and Cloudflare build don't break.
- Chorus-specific assets are **generated** from `schema/tokens/` and ported routes/components under `src/routes/` and `src/components/`.
- Sync direction is **chorus → lovable, one way**. Anything Lovable's editor adds on the other side should live under a path we don't overwrite.

## Regenerate

From the repo root:

```bash
npm run build:lovable
```

This pipeline runs `schema/tokens/build-resolved.mjs` (resolves token JSON + `tokens.css`) then `scripts/build-lovable.mjs`, which:

1. Writes `lovable-export/src/styles.css` — Lovable's shadcn baseline `@theme` block + appended chorus tokens, with the `data-theme="dark"` block mirrored onto `.dark` so Tailwind's dark variant and chorus dark stay in sync.
2. Syncs `lovable-export/docs/` — `manifest.json`, `catalog.md`, `DESIGN.md`, `screens/`, `components/<family>/{md,spec.json,family.json}`, `tokens/{reference,system,component,resolved.*}.json`, `patterns/*.{md,png}`, `icons/README.md`, `CONSUMING.md`.
3. Syncs `lovable-export/src/components/chorus/` — ports `packages/ui/src/*` from JSX→TSX (Lovable's idiom): `.jsx`→`.tsx`, `.js`→`.tsx` if it contains JSX else `.ts`, with relative-import extensions stripped and a `// @ts-nocheck` banner on each TS output. CSS passes through.

The whole sync is idempotent and wipes target subtrees each run — anything removed from chorus disappears here too.

The pipeline runs automatically in two places:

- **Local pre-commit** ([`.githooks/pre-commit`](../.githooks/pre-commit), wired via `npm run prepare` setting `core.hooksPath`) — when a commit stages files under `schema/`, `patterns/`, `packages/ui/src/`, or `scripts/build-lovable.mjs`, the hook re-runs `npm run build:lovable` and stages any resulting `lovable-export/` diff into the same commit. Bypass with `--no-verify` only when intentional.
- **CI stale check** ([`.github/workflows/ci.yml`](../.github/workflows/ci.yml), `lovable-export` job) — runs `build:lovable` on every PR/push and fails if the export would change. Catches drift if someone bypassed the local hook.

## Copy to the Lovable repo

Lovable only syncs via GitHub, so the handoff is a manual overlay. This folder's layout matches the Lovable-generated GitHub repo template 1:1, so the import is a single drag — see [`IMPORT.md`](IMPORT.md) for the procedure.

In short: select every entry inside `lovable-export/` (configs, `src/`, `public/`, `docs/`, dotfiles), drag onto the cloned Lovable repo's root, choose **"Replace All"**. Then in the target repo:

1. `git status` — review what changed.
2. `bun install` only if `package.json` differs.
3. Commit and push.

### Files chorus owns (safe to overwrite)

Root configs, `AGENTS.md`, `PROMPTING.md`, `docs/`, and `src/components/chorus/` are fully chorus-managed.

### Files in the shared zone (watch for conflicts)

- `src/routeTree.gen.ts` — auto-generated by TanStack Router; if Lovable added routes, this will differ. Let Lovable regenerate it after the import.
- `package.json` — replaced wholesale; diff first if Lovable added deps.

## Scope notes

- Tokens flow through as `var(--sys-*)` / `var(--ref-*)` plus a mirrored `.dark` block so Tailwind's dark variant matches our dark tokens.
- shadcn semantic tokens (`--primary`, `--background`, ...) are kept verbatim as a Tailwind boot baseline only — they are NOT the design contract. Author against chorus tokens, never against them.
- The current export ships **only the index route**. Additional routes (`components/*`, `tokens/*`, etc.) will be added as we port them.
